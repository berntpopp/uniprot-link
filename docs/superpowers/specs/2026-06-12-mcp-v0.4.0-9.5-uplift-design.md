## 0. TL;DR

v0.3.0 is genuinely good (8/10): uniform envelope, release pinning, citation per
call, read-only enforcement, caching as mode-projection, `next_commands` chaining,
HGVS-style variant notation. The residual gap to 9.5 is **not architecture** ŌĆö it
is **contract consistency** (the error envelope diverges across tools), **two
genuine data bugs** (the feature-type filter vocabulary is incomplete; the disease
"description" is the wrong field), **per-call token overhead**, and **one latency
hotspot** (`find_proteins`). Plus two forward-looking ergonomics wins the user has
opted into: **typed structured output** (MCP 2025-06-18) and **content-aware
chaining**.

The work is grouped into ten deltas (AŌĆōJ) below. Each names the bug(s) it closes,
the verified root cause, the validated fix, and the scoring dimension it lifts.

## 1. Two corrections to the assessment (why we validate live)

The assessment is careful and its reproductions are accurate, but two of its
*proposed fixes* rest on assumptions the live data contradicts:

1. **Bug 9 (disease definition).** The assessment proposed reading
   `skos:definition` "available in the diseases graph." **It is not present.**
   Live probe of `<http://purl.uniprot.org/diseases/4356>` (AOA4) returns no
   `skos:definition`; the real human-readable definition is on the disease via
   **`rdfs:comment`**: *"An autosomal recessive disease characterized by cerebellar
   ataxia, oculomotor apraxia, areflexia and peripheral neuropathy."* The fix is
   correct in spirit, wrong in predicate ŌĆö we join `?disease rdfs:comment`, not
   `skos:definition`.

2. **Bug 11 (SPARQL syntax detail).** The assessment expected "QLever usually
   reports a parse position." For the probed malformed queries QLever returned a
   **400 with an empty body**, so the client correctly fell back to its generic
   string. Passing through a parse position only helps *when the endpoint supplies
   one*; we surface the body when present and otherwise give a structured,
   cause-oriented hint + recovery command. We do **not** promise a parse position
   the endpoint does not return.

Everything else in the assessment reproduced exactly (verified live this session):
`feature_types:["natural_variant"]` ŌåÆ `invalid_input`; the allowed-list cut at
`"...np_binding, pep"`; `compact` sequence == full (521-aa string returned);
`get_taxon("Homo sapiens")` missing `elapsed_ms`/`cached`/`next_commands`/`rank`;
disease `description` identical boilerplate for both AOA4 and MCSZ; GO terms with
no evidence codes.

## 3. The v0.4.0 deltas

Every SPARQL below was validated live this session; p50 single-call timings noted.

### A. Error-envelope uniformity & richness ŌĆö *error handling 7 ŌåÆ 9ŌĆō10*

The error envelope is the single highest-ROI consistency lift: an LLM optimizes
against the *shape* of failure.

**A1 ŌĆö Structured `allowed_values`, never truncated (Bug 2).**
- Extend `InvalidInputError` with optional structured fields:
  `field: str | None`, `allowed: list[str] | None`, `hint: str | None`.
- The error envelope (`_error_envelope`) surfaces these as top-level keys when
  present: `field`, `allowed_values` (full array ŌĆö never in the message),
  `hint`, and `capabilities_ref` (e.g. `"feature_types"`) pointing at the
  capabilities key that lists the full vocabulary.
- Keep `_safe_message` capped (messages stay short) but move the *data* (the
  list) out of the prose. `protein_features` raises
  `InvalidInputError(msg, field="feature_types", allowed=sorted(FEATURE_TYPES))`.
- The message itself stops embedding the list: `"Unknown feature type 'X'. See
  allowed_values."`.

**A2 ŌĆö `next_commands` on *every* error (Bug 3).**
- Give the envelope a **default** recovery when no explicit `fallback` is set, so
  uniformity is structural, not per-call discipline:
  - `not_found` / `invalid_input` on a `get_protein*` tool ŌåÆ suggest
    `find_proteins` (sanitized, see G2) and `get_server_capabilities`.
  - any error with no better idea ŌåÆ `get_server_capabilities`.
- Add explicit, *useful* per-tool fallbacks where one exists (e.g.
  `get_protein_features` invalid_input ŌåÆ `get_server_capabilities` to read the
  feature vocabulary; `get_taxon` not_found ŌåÆ `find_proteins`/`search_example_queries`).
- Builder: `default_error_next_commands(tool, error_code, args)` in
  `next_commands.py`; the envelope calls it whenever `context.fallback is None`.

**A3 ŌĆö Fix the wrong tool name (Bug 4).** `queries.py` `find_proteins` anchor
error: `sparql_query` ŌåÆ `run_sparql_query`. (Also audit all instruction/resource
strings for the same typo.)

**A4 ŌĆö `get_taxon` by-name parity (Bug 5).**
- By-name path uses `_select_timed`; payload carries `elapsed_ms` + `cached`.
- `resolve_taxon_by_name` adds `?rank` (and shaping surfaces it).
- `taxonomy.py` adds `next_commands` for the by-name shape: for the top match,
  `get_taxon(id=<top>)` and `find_proteins(organism_taxon=<top>)`. The
  nameŌåÆidŌåÆfind chain no longer dead-ends.

**A5 ŌĆö Actionable `query_syntax_error` (Bug 11).**
- When the QLever body is non-empty, surface it (unchanged).
- When empty, message becomes cause-oriented:
  *"The endpoint rejected the query as malformed (no detail returned). Common
  causes: unbalanced `{}`/`()`, a missing PREFIX, or an incomplete FILTER. Re-seed
  from a working example."* + the existing `search_example_queries` recovery
  command. No false promise of a parse position.

### B. Feature-type contract ŌĆö *correctness + schema design (Bug 1)*

**Root cause:** the filter vocabulary is incomplete, so the emit path
(`shape_features` fallback) produces keys the filter path rejects. The fix makes
**emit ŌŖå filter by construction.**

- **Complete the registry.** Enumerate the full set of UniProt feature-annotation
  classes that carry a `faldo` range, validated against the `up:` core ontology
  live, and add the missing ones to `FEATURE_TYPES`. Confirmed-missing from the
  current 28: `natural_variant` (`Natural_Variant_Annotation`),
  `alternative_sequence` (splice/`Alternative_Sequence_Annotation`),
  `sequence_conflict` (`Sequence_Conflict_Annotation`). Also add the residue-status
  classes that can appear with ranges: `non_terminal_residue`,
  `non_adjacent_residues`, `initiator_methionine`, `unsure_residue`,
  `non_standard_residue` (each validated to exist before inclusion).
- **Make divergence impossible going forward.** `shape_features` must emit **only**
  registry keys. Change the fallback: if a class is *not* in
  `FEATURE_CLASS_TO_KEY`, emit a namespaced raw key `"_unmapped:<ClassLocalName>"`
  and add a payload-level `unmapped_types: [...]` note ŌĆö so an unfilterable type is
  *visibly* unfilterable rather than silently presenting a key that fails. (In
  practice, with the registry completed, this path should never trigger for the
  validated corpus.)
- **Capabilities + description sync.** `capabilities.feature_types` already derives
  from `FEATURE_TYPES`, so completing the registry fixes it automatically. Keep the
  description's "each returned `type` round-trips" promise ŌĆö now true.
- **Regression test:** for a fixed entry set covering the previously-unmapped
  classes (Q96T60 emits `natural_variant`/`alternative_sequence`/`sequence_conflict`),
  assert every emitted `type` is in `capabilities.feature_types` and re-filtering on
  it returns Ōēź1 row.

### C. Token efficiency ŌĆö *token 6 ŌåÆ 9; chaining 8 ŌåÆ 9*

**C1 ŌĆö Windowed `compact` sequence (Bug 6).** Re-grade `get_sequence`:
- `minimal`: metadata only (`isoform`, `canonical`, `length`, `mass_da`) ŌĆö no
  residues. *(unchanged)*
- `compact` (**default**): metadata + `sequence_preview` (first 30 + `ŌĆ”` + last 30
  residues) + `sequence_truncated: true`. **No full string.**
- `standard`: full `sequence` string.
- `full`: full `sequence` + raw isoform IRIs.
- Document the grade in the tool description and `uniprot://usage`. For titin
  (~34,350 aa) the default call drops from tens of KB to ~80 chars.

**C2 ŌĆö `_meta` diet.** Reduce fixed per-call overhead without losing grounding:
- Trim `next_commands` from 4 ŌåÆ the **top 2** highest-value steps.
- Drop `endpoint` from per-call `_meta` (static; in capabilities + the success
  payload's existing context). **Keep** `uniprot_release`, `unsafe_for_clinical_use`,
  and the short `citation` DOI (cheap, high grounding value ŌĆö the assessment scored
  grounding 9 *because* of these).
- Net: ~50% fewer `_meta` bytes per call; `minimal` becomes genuinely lean.

**C3 ŌĆö Content-aware `next_commands` (chaining).** Where a tool already knows
counts, suggest only non-empty sub-resources:
- `get_protein_diseases` (diseases present) ŌåÆ lead with `get_protein_variants`
  (disease-linked evidence) then `get_protein`.
- `get_protein_features`/`get_protein_variants` ŌåÆ if `count == 0`, suggest
  `get_protein` + `get_server_capabilities` instead of a dead sibling.
- `find_proteins` ŌåÆ first `get_protein` for the top hit, then (only if >1 hit)
  paginate. Keep `after_get_protein` static but trimmed to the 2 most-used
  sub-resources (`sequence`, `features`).

### D. `find_proteins` latency ŌĆö *latency 6 ŌåÆ 8ŌĆō9 (Improvement #3)*

**Root cause:** `ORDER BY DESC(?reviewed) ?mnemonic` materializes + sorts the full
pre-LIMIT match set; broad anchors (keyword/EC) match across 250M-row TrEMBL.

**Design ŌĆö reviewed-first, bound, small-set sort:**
- When `reviewed is None` (the default): **query Swiss-Prot first**
  (`up:reviewed true`, `ORDER BY ?mnemonic`, bound + small), LIMIT/OFFSET on that
  segment. Swiss-Prot (~570k) sorts fast; results are also the curated, more useful
  hits.
- **Spill** to TrEMBL only when the reviewed segment is exhausted for the requested
  page (deep pagination or a TrEMBL-only anchor). Determine the boundary with a
  cheap bound `COUNT` on the reviewed segment (validated fast on anchored sets) and
  slice OFFSET across the two segments.
- When `reviewed` is explicitly `true`/`false`: single bound query (already fast).
- Drop the global `DESC(?reviewed)`; each segment is `ORDER BY ?mnemonic` over a
  bound set, with the final page assembled reviewed-then-unreviewed in Python.
- **Re-validate** every anchor (gene+taxon, keyword, EC) with
  `research/verify_queries.py` against the live endpoint; record before/after p50.
  Acceptance: keyword/EC default page < 3 s (from 6ŌĆō9 s); no behavior regression
  on the integration suite. If the COUNT/slice proves costly, fall back to the
  simpler "reviewed-only default page, explicit opt-in for TrEMBL" variant (still
  kills the hotspot) and document it.

### E. GO evidence codes ŌĆö *grounding/citation (Bug 10)*

**Validated** (293 ms): evidence is reified ŌĆö
`?stmt rdf:subject <entry>; rdf:predicate up:classifiedWith; rdf:object ?go;
up:attribution ?attr . ?attr up:evidence ?eco` ŌåÆ ECO IRIs (e.g. `ECO_0000314`).

- Extend `protein_go_terms` with a **LEFT JOIN** to the reified evidence so terms
  without a statement still appear; aggregate distinct ECO per `?go`.
- Per term, add `evidence: ["ECO:0000314", ...]` and a friendly
  `evidence_codes: ["IDA", ...]` via a small static **ECOŌåÆGO-evidence-code** map for
  the common codes (IDA/IEA/IMP/IPI/ISS/IBA/TAS/NAS/ŌĆ”); unmapped ECO ids pass
  through as the raw `ECO:...`.
- Validate full-entry timing (all terms + evidence) stays < 1 s; if the reified
  join regresses on annotation-dense entries, gate evidence behind
  `include_evidence: bool = True` with a fast no-evidence path.

### F. Disease definition ŌĆö *grounding/citation (Bug 9)*

**Validated:** disease definition lives on `?disease rdfs:comment`; `skos:definition`
absent. Restructure `protein_diseases` output for honesty:
- `involvement`: the annotation's `rdfs:comment` (the "caused by variants
  affectingŌĆ”" note) ŌĆö renamed from the misleading `description`.
- `definition`: the disease vocabulary entry's `rdfs:comment` (the real clinical
  definition).
- add `mnemonic` (e.g. `AOA4`) from `?disease up:mnemonic`.
- keep `disease`, `disease_id`, `mim`.
- Update the tool description to name both fields and their distinction.

### G. Accession validation & recovery ŌĆö *Bugs 7, 8*

**G1 ŌĆö Official accession pattern (Bug 7).** Replace `_ACCESSION_RE` with the
documented UniProtKB grammar so malformed input fails locally:
`^([OPQ][0-9][A-Z0-9]{3}[0-9]|[A-NR-Z][0-9]([A-Z][A-Z0-9]{2}[0-9]){1,2})(-\d+)?$`
(case-insensitive). `999999` ŌåÆ cheap `invalid_input` (no round-trip); `P05067`,
`P05067-2`, `A0A024R1R8` still pass. Error carries an example + `field`.

**G2 ŌĆö Sanitized not_found recovery (Bug 8).** `get_protein` fallback must not
reuse a non-gene accession as `gene`. Build the recovery context-aware: suggest
`find_proteins(gene=<value>)` **only** when `<value>` matches a gene-symbol shape
(alpha-leading, Ōēż 12 chars); otherwise suggest `get_server_capabilities` +
`search_example_queries`. Applies to the `get_protein` and `get_protein_*` family.

### H. Example-catalog discovery polish ŌĆö *Bug 12 / Improvement #6*

- `search_example_queries`: `GROUP BY ?ex ?comment` only; collapse `?type` with
  `SAMPLE`; an example never appears twice.
- Rank UniProt-native examples above federated (Rhea/empty-keyword) ones: add a
  `?federated` flag (e.g. via `EXISTS` of a `federatesWith` or a Rhea graph
  reference) and `ORDER BY ?federated ?ex`, or post-sort in shaping.
- Drop or de-emphasize empty-keyword hits in ranking; never crash on empty.
- Validate the dedup live (the prior 126ŌåÆ count should hold with no dup ids).

### I. Typed structured output ŌĆö *schema design 9 ŌåÆ 10; consistency 8 ŌåÆ 10 (Improvement, user opt-in)*

Adopt the MCP 2025-06-18 `outputSchema`/`structuredContent` contract via FastMCP
3.4.2.

- For each **typed** tool (`get_protein`, `get_protein_sequence`,
  `get_protein_features`, `get_protein_variants`, `get_protein_diseases`,
  `get_protein_cross_references`, `get_protein_go_terms`, `map_identifiers`,
  `find_proteins`, `get_taxon`, `get_server_capabilities`), declare an
  `output_schema` describing the **success** payload.
- Schemas live in a new `uniprot_link/mcp/schemas.py` (keeps tool modules under the
  600-line cap). They are **permissive**: optional fields (response_mode projection
  drops some), `_meta` object, and `additionalProperties: true` so the envelope's
  `success`/error keys never violate the schema. The schema is documentation +
  client-side typing, not a straitjacket.
- `run_sparql_query` keeps a **generic** schema (columns are dynamic): a loose
  object with `query_type`, `columns`, `rows`, `boolean`, `data` all optional.
- Verify FastMCP 3.4.2 emits `structuredContent` **and** a back-compat
  `TextContent` JSON block (the spec's SHOULD); add an MCP-layer test asserting
  both are present for a representative tool.
- If FastMCP 3.4.2's output-schema API cannot accommodate the dynamic envelope
  cleanly, fall back to annotating return types on the success payloads only and
  document the limitation ŌĆö structured output is the goal, not a specific wiring.

### J. Request id ŌĆö *observability 9 ŌåÆ 10*

Add a per-call `request_id` (uuid4 hex, short) to every `_meta` (success and
error) in `run_mcp_tool`, so a multi-call session is correlatable in logs and by
the consumer. Log it alongside the existing `mcp_tool_error` line.

## 4. Explicitly out of scope (with reasons)

- **Feature-level evidence / InterPro / Pfam *on a domain extent*.** Confirmed
  absent in the RDF in the prior milestone (a `Domain_Extent_Annotation` carries
  only range/type/comment). GO evidence (E) is different and *is* present ŌĆö do not
  conflate.
- **Isoform mass backfill.** UniProt data gap; computing mass from sequence is
  error-prone (PTMs). Surface UniProt's value or `null`.
- **`subClassOf*` transitive GO/taxonomy paths.** Time out on QLever; the
  single-hop approach stays.
- **Capabilities content-hash** (hnf1b-style `capabilities_version`). Nice, but not
  needed for >9.5; defer to a later polish unless trivial alongside I.

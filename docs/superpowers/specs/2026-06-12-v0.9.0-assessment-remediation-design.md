## Research grounding (2025–2026 MCP best practice)

Confirmed by web research (`anthropic.com/engineering/writing-tools-for-agents`,
MCP spec 2025-06-18, FastMCP docs) and live endpoint probes:

1. **Silent-empty for a valid-but-redirected identifier is the worst failure mode**
   for an evidence-grounding tool. The fix is one of: normalize + return with an
   explicit note, **or** an explicit typed error — never empty-as-fact. (Directly
   governs F1/F2.)
2. **`up:mass` exists ONLY on the canonical `-1` isoform** (live-verified on
   P05067 and P38398). Alternative-isoform mass must be *computed*, never required
   or defaulted. The existing `shape_sequences`/`average_mass` already does this —
   the F2 fix must preserve it.
3. **Canonical isoform = IRI ending in `-1`** (confirmed). `up:mnemonic "X"` is a
   fast bound anchor (confirmed). Validates the F3 mnemonic fast-path.
4. **Prefer omitting absent fields over null; keep output schemas permissive.**
   The repo already follows this; new fields (`reviewed_count`, `requested_accession`,
   `serialization`) are additive and optional.
5. **`next_commands` is a sound project convention, not an MCP standard** — keep
   it, do not regress it.

## Findings → changes

### F1 (HIGH) — `get_protein_features` silent-empty for isoforms + the latent twin

**Root cause:** `queries/proteins.py::protein_features` anchors on
`uniprotkb:{validate_accession(accession)}` **without** `.split("-")[0]`, so
`P05067-2` anchors a non-existent feature subject → 0 rows. The same un-stripped
anchor is in `protein_cross_references` (→ `map_identifiers`) and
`protein_go_terms`. `variants`/`diseases` already strip correctly.

**Change:**
- Strip the isoform suffix to the base entry in `protein_features`,
  `protein_cross_references`, and `protein_go_terms` (use the existing
  `validate_accession(accession).split("-")[0]` idiom — same as `protein_variants`).
- Make isoform handling **consistent across the whole `get_protein_*` family**
  (the assessment's core ask). Centralize in the service: `require_entry` becomes
  `resolve_entry`, returning the `EntryStatus`, and now **rejects a typo'd isoform
  index** (`isoform_exists is False`) exactly as `get_protein` does — so
  `get_protein_features(P05067-99)` is a clean `not_found`, never canonical data
  under a wrong label.
- When a **valid** isoform suffix was supplied, every entry-level sub-tool
  (features, variants, diseases, go_terms, cross_references, map_identifiers)
  attaches `requested_accession` + an `isoform_note` stating the annotations are
  entry-level for the canonical accession — matching `get_protein`'s model.

### F2 (HIGH) — `get_protein_sequence` returns `not_found` for valid isoforms

**Root cause:** `protein_sequence` anchors on the full `uniprotkb:{acc}` (incl.
`-N`); `up:sequence` hangs off the base entry only → 0 rows → `not_found`.

**Change:**
- `protein_sequence` anchors on the **base** entry (`.split("-")[0]`), returning
  all isoforms (live-verified: P38398 → `-1..-8`, mass only on `-1`).
- `get_sequence` service: when an isoform suffix was supplied, return **that
  specific isoform** as the primary `canonical` record (rename intent: it is the
  *requested* isoform), set `requested_isoform`, and reject if the index does not
  exist (consistent typed `not_found`). The data stays isoform-specific (its own
  length + computed mass) — **not** collapsed to canonical, per the assessment.
- Fixes the broken cross-tool contract where `get_protein(P05067-2)` advises
  calling `get_protein_sequence` for the isoform sequence.

### F3 (MEDIUM) — exact-key `find_proteins` runs as cold scans

**Change (two safe, validated wins):**
1. **Mnemonic fast-path:** a mnemonic uniquely identifies ≤1 entry, so when
   `mnemonic` is the anchor and `reviewed` is unset, issue **one** bound query
   (no two-phase reviewed-first, no redundant TrEMBL scan). Live-validated:
   7891 ms → ~4500 ms, and it removes the wasteful unreviewed scan leg entirely.
2. **Concurrent reviewed-first legs (offset 0):** for the common `offset==0` page,
   issue the reviewed COUNT, reviewed fill, and (speculative) unreviewed fill
   **concurrently** via `asyncio.gather` instead of sequentially. Wall-clock drops
   from sum→max of the legs (the gene "no reviewed" worst case 10.4 s → ≈ slowest
   single leg). Semantically identical: reviewed-first ordering and paging
   preserved. `offset>0` keeps the existing sequential path.

The deeper single-gene-join cost (~6 s on QLever) is left to the endpoint; per
AGENTS.md QLever discipline we do **not** add an `ORDER BY` or restructure the
join in ways that risk a 45-min timeout. Timing re-validated with
`research/verify_queries.py`.

### F4 (MEDIUM) — `latency_profile` under-states two tools

**Change:** Re-calibrate honestly from this session's cold measurements. Move
`get_protein_features` and `get_protein_diseases` out of `fast` (0–700 ms) into a
`medium` band (~700–2500 ms). Reframe `medium`/`slow_cold_scan` so bands describe
*query class* (bound lookup vs. FALDO/aggregation vs. unbounded scan), as best
practice recommends, rather than a single number the server will mis-predict.

### F5 (MEDIUM) — `default_select_limit: 50` contradicts find_proteins paging

**Change:** In `capabilities.limits`, keep `default_select_limit` (it is the
`run_sparql_query` auto-LIMIT) but **clarify its scope in a note**, and add
`find_proteins_page_size: 25`, `find_proteins_max_limit: 200`, and
`cross_reference_compact_id_cap: 25` so every advertised number matches behavior.

### F6 (LOW) — `search_example_queries` lacks a `query`/`q` alias

**Change:** Add `"query": "text"` and `"q": "text"` to `ARG_ALIASES`. The
per-tool alias guard already ensures `query=` still resolves to `query` on
`run_sparql_query` (canonical wins) and only rewrites to `text` where `text` is a
real param (the search tool). Reverse-doc map surfaces it in capabilities.

### F7 (LOW) — no canonical-only sequence mode (token sink)

**Change:** Add `canonical_only: bool = False` to `get_protein_sequence`. When
true, return only the canonical record and omit the `isoforms` list (count still
reported). Composes with `response_mode` (e.g. `standard` + `canonical_only` =
just the full canonical string).

### F8 (LOW) — CSV/TSV SELECT mislabeled `query_type: "RDF/raw"`

**Change:** `run_query` already classifies the operation (`classify_sparql_operation`).
Report the **true** query form as `query_type` for non-JSON serializations
(SELECT/ASK/CONSTRUCT/DESCRIBE) and add a separate `serialization` field
(`csv`/`tsv`/`turtle`/…). JSON SELECT/ASK labels are unchanged.

### F9 (LOW) — gene-anchor page dominated by TrEMBL noise

**Change:** Surface `reviewed_count` on `find_proteins` whenever the reviewed-first
path ran (`reviewed is None`) — it is the `cr` the two-phase pagination already
computes, threaded out at no extra cost. When unreviewed entries dominate, add a
one-line `reviewed_hint` suggesting `reviewed=true`. Non-breaking; no default flip
(a gene present only in TrEMBL must still be findable).

### Untested — obsolete / demerged accession regression coverage

The obsolete path is already implemented (`entry_status`,
`build_obsolete_record`, `ObsoleteEntryError`). Add **regression tests**: a unit
test (mocked) for `get_protein` obsolete + a sub-tool obsolete `not_found`, and an
`@pytest.mark.integration` test against a real obsolete and a real demerged
accession.

- F9: `find_proteins(gene=…)` (reviewed unset) carries `reviewed_count`.

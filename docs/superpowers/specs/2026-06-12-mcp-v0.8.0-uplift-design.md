# uniprot-link v0.8.0 — LLM-Consumer Uplift Design

**Date:** 2026-06-12
**Author:** MCP engineering pass (driven by `docs/mcp-assessment-v0.7.0-tester.md`)
**Goal:** Close every defect in the v0.7.0 tester assessment (F1–F8 + Part 1 UX
items) so an LLM consumer rates the server **> 9.5 / 10**. No core-correctness
defect was found in v0.7.0; the issues cluster in (a) enum-value discoverability,
(b) silent/misleading edges, and (c) felt latency. This design targets all three.

Research backing (official MCP spec 2025-06-18, Anthropic "Writing effective
tools for agents" + "Code execution with MCP", FastMCP docs, Datadog production
lessons) is summarized inline per fix; the through-line is: **surface valid
values, never return a false-empty, keep one consistent envelope, trim default
tokens.**

---

## Scope

In scope: the 8 findings (F1–F8) and the three Part-1 UX levers (cold gene-search
latency, leaner `get_protein_features` default, redundant echo). Out of scope:
SPARQL data-model changes, new graphs, auth, anything not traceable to the
assessment.

Design principle: **additive and pattern-preserving.** Reuse the existing error
envelope, `next_commands` chaining, `response_mode` ladder, shaping helpers, and
the QLever discipline (bound joins, sort small sets in Python). Respect the
600-line module cap (`make lint-loc`).

## F1 — Invalid enum *value* returns argument *names* (HIGH)

**Root cause.** `ArgValidationMiddleware.on_call_tool` catches *every* pydantic
binding `ValidationError` — including `Literal`/enum *value* failures
(`aspect="function"`, `detail="banana"`, `result_format="banana"`,
`response_mode="banana"`) — and routes them through `build_arg_error_envelope`,
which unconditionally puts the **argument names** in `allowed_values` and prints
"Valid argument **names** are listed in allowed_values." Wrong category for a
value error; the valid values appear nowhere.

**Fix (two parts).**

1. *Error path.* In the middleware, classify the binding error into one of three
   categories from the pydantic error `type` + `loc`:
   - `missing` → missing required arg (loc is a known param, type is `missing`/
     `missing_argument`). Keep the arg-names wording.
   - `unknown_name` → loc is **not** a known param. Keep arg-names + did-you-mean.
   - `bad_value` → loc **is** a known param but the value failed (e.g.
     `literal_error`, `enum`, `greater_than_equal`). New branch.

   For `bad_value`, extract the field's allowed value set from the cached JSON
   input schema via a new pure helper `enum_values_for(schema, param)` (handles a
   direct `enum` and an `anyOf`/`oneOf` branch carrying `enum`, e.g.
   `Literal[...] | None`). Then `build_arg_error_envelope` emits:
   - message: *"Invalid value for argument `aspect` of get_protein_go_terms.
     Valid values are listed in allowed_values."* (category: value, not names)
   - `field = aspect`, `allowed_values = [<enum values>]`, `hint = <signature>`.
   - If the field has no enum (a numeric `ge/le` constraint), fall back to
     *"Invalid value for argument `limit`: <pydantic reason>."* with no
     `allowed_values` (don't fabricate a list); keep `hint = signature`.

   This matches the research's #1 takeaway: an invalid enum must return the
   *field's valid values* + a self-correcting message, surfaced in a result the
   LLM can read (this server's structured envelope is the `isError:true`
   equivalent).

2. *Discoverability before a failed call.* Add `argument_value_sets` to
   `get_server_capabilities` (and `uniprot://capabilities`): a
   `{tool: {arg: [values]}}` map computed from the live schemas (new
   `collect_tool_enums(mcp)` mirroring `collect_tool_signatures`). FastMCP already
   emits `Literal` as JSON-Schema `enum` in each tool's `inputSchema`; this echoes
   those sets into the discovery surface so an agent sees `aspect`/`detail`/
   `result_format`/`response_mode` values without provoking an error first.

**Files:** `mcp/arg_help.py` (new `enum_values_for`), `mcp/middleware.py`
(categorize + pass enum values), `mcp/envelope.py` (`build_arg_error_envelope`
value branch), `mcp/capabilities.py` (+`argument_value_sets`,
`collect_tool_enums`), `mcp/tools/discovery.py` (pass enums through).
**Tests:** `test_arg_middleware.py`, `test_arg_help.py`, `test_capabilities.py`,
`test_discovery_surface.py`.

## F2 — `get_protein_cross_references` silently returns empty for unknown DB (HIGH)

**Root cause.** `databases=["NOTADB"]` builds a `VALUES ?db { <…/database/NOTADB> }`
filter → zero rows → `database_count:0, counts:{}` with no signal. Reads like a
valid "no data" answer (the assessment's most dangerous failure mode).

**Design decision — echo, don't hard-reject.** UniProt has ~180 cross-reference
databases; only 20 are in `COMMON_XREF_DATABASES`. Hard-validating against the
curated list would *false-reject* legitimate databases. So mirror the
`map_identifiers` contract (which the assessment praised): always echo
`requested_databases`, and compute `unmatched_databases` = requested names that
produced **zero** rows. Add a `database_hint` when any are unmatched, naming
case-sensitivity and pointing at `common_xref_databases` in capabilities, plus a
`did_you_mean` against the common set for close misses. This makes a typo
unmistakable (`unmatched_databases: ["NOTADB"]`) without risking false rejects —
satisfying the research's "kill false-empty / echo requested vs matched" guidance
while staying safe for the open DB vocabulary.

`map_identifiers` calls `get_cross_references`, so it inherits
`unmatched_databases` for free (its default-DB set that a given protein lacks is
now disclosed too).

**Files:** `services/sparql_service.py` (`get_cross_references` computes
unmatched), `services/shaping.py` (helper for did-you-mean if placed there),
`mcp/schemas.py` (CROSS_REFERENCES_SCHEMA + MAP_IDENTIFIERS_SCHEMA add
`requested_databases`/`unmatched_databases`). **Tests:**
`test_service_and_tools.py`, `test_shaping.py`.

## F3 — `get_taxon` name scan: exact match not first; chain points at a hybrid (HIGH)

**Root cause.** `resolve_taxon_by_name` filters `exact || contains` and orders
`ORDER BY ?scientificName` (alphabetical), so `Takifugu rubripes` (exact, id
31033) lands at position 5 behind hybrids/viruses; `next_commands[0]` then chains
to a hybrid (2506766).

**Fix.** Per the QLever "sort small sets in Python" discipline: the scan returns
≤10 rows, so re-rank in `SparqlService.get_taxon` after `shape_taxon_resolutions`,
using the query string. Ranking key (ascending = best first):
1. exact `scientific_name` == query (case-insensitive) →
2. exact `common_name` == query →
3. `scientific_name` startswith query →
4. substring; tie-break: non-hybrid (no " x ") before hybrid, then shorter
   `scientific_name`, then alphabetical.

`matches[0]` becomes the best match, so the existing `next_commands`
(`get_taxon(top)` → `find_proteins(organism_taxon=top)`) now points at the right
organism. Tag the top match `match_quality: "exact"` when rank 0/1.

**Files:** `services/sparql_service.py` (rank in `get_taxon`), or a pure
`rank_taxon_matches(matches, query)` in `services/shaping.py` (preferred — unit
testable). **Tests:** `test_shaping.py`, `test_service_and_tools.py`.

## F4/F5 — Truncation envelope: wrong `total`, inconsistent shape (MEDIUM)

**F4 root cause.** `get_features` runs SPARQL `LIMIT {limit}`, so `all_features`
is itself capped at `limit`; `truncated.total = len(all_features) = limit`, not
the true count.

**F5.** Five tools use four different truncation shapes.

**Fix — one standard shape `{returned, total?, reason, recovery}`** (omit `total`
only where a true count is genuinely not cheaply computable), applied as:

| Tool | How `total` is obtained |
|------|--------------------------|
| `get_protein_features` | Decouple the SPARQL fetch cap (`_FEATURE_FETCH_CAP = 1000`, the existing max) from the display `limit`; slice to `limit` in Python; `total = len(all_features fetched)`. Mirrors the proven `get_protein_go_terms` "fetch-at-cap, cap-in-Python" pattern. |
| `get_protein_go_terms` | Already has `{returned, total, recovery}`; **add `reason`**. |
| `get_protein_variants` | Add a cheap parallel `COUNT(DISTINCT ?a)` over `Natural_Variant_Annotation` (new `protein_variants_count` builder, run in the existing `asyncio.gather`); `total` = exact variant count. |
| `find_proteins` | On truncation only, one extra bound `COUNT` query (existing `find_proteins(count=True)` builder, honoring the user's `reviewed` filter); `total` = grand match count. Extra cost paid only when the caller is actually paging. |
| `run_sparql_query` | Arbitrary query: `total` not computable without re-running. Standardize to `{returned, reason, recovery}` (add `returned = row_count`); document the deliberate `total` omission. |
| xref compact per-DB | Already `{returned, total}` per database — leave; it is already on-pattern. |

**Files:** `services/sparql_service.py` (all four tools),
`services/queries/proteins.py` (new `protein_variants_count`; features fetch
cap), `services/shaping.py` (truncation already shaped inline — keep). **Tests:**
`test_service_and_tools.py`, `test_queries.py`, `test_structured_output.py`.

## F6 — `name_contains` is a literal substring (MEDIUM)

**Root cause.** `FILTER(CONTAINS(LCASE(?name), LCASE("polynucleotide kinase")))`
misses "Bifunctional polynucleotide **phosphatase/**kinase".

**Fix — per-word AND CONTAINS.** Split `name_contains` on whitespace (cap at 6
tokens, escape each); require every token to appear somewhere in the name:
`FILTER(CONTAINS(LCASE(?name),"polynucleotide") && CONTAINS(LCASE(?name),"kinase"))`.
Single-word input behaves exactly as before; multi-word now matches the realistic
case. `name_contains` is always paired with the bound `organism_taxon` anchor, so
the filter runs over a bounded set (QLever-safe). Document the new semantics in
the tool description, server instructions, and capabilities. **Re-validate live**
with `research/verify_queries.py` per CLAUDE.md (PNKP in human: taxon 9606 +
`name_contains="polynucleotide kinase"` must now return Q96T60).

**Files:** `services/queries/proteins.py` (`find_proteins` name pattern + count
join), `mcp/tools/proteins.py` (description), `mcp/resources.py` (instructions),
`mcp/capabilities.py` (note). **Tests:** `test_queries.py` + a live integration
case.

## F7 — `requested_accession` echoed even when identical (LOW)

**Fix.** In `get_protein` (and the obsolete-record path), include
`requested_accession` **only** when the user's input differs from the normalized
base accession (`accession.strip().upper() != acc`) — i.e. on an isoform
(`Q96T60-1` → `Q96T60`) or redirect. Pure token win on the most-called tool.
**Files:** `services/sparql_service.py`. **Tests:** `test_service_and_tools.py`,
`test_structured_output.py`.

---

## F8 — `run_sparql_query` success omits `next_commands` (LOW)

**Fix.** Add `_meta.next_commands` to `run_sparql_query` success via a new
`after_run_sparql(payload)` builder: if a SELECT row carries a recognizable
UniProt accession/entry IRI → `get_protein(accession=…)`; else
`search_example_queries`. Restores the universal chaining contract (framed in the
closure record as a community convention, not a spec mandate, per research #7).
**Files:** `mcp/next_commands.py`, `mcp/tools/query.py`. **Tests:**
`test_next_commands.py`, `test_service_and_tools.py`.

## Part 1 — Felt latency + token noise

### P1a — Cold multi-gene latency → concurrent batch resolver (new tool)

**Why not a static gene cache.** A curated gene→accession map (the `COMMON_TAXA`
analogue) would drift from the UniProt release and risk mis-grounding. The
grounding-safe lever the assessment itself suggests ("add a batch form so N genes
do not cost N×5 s") is **concurrency**.

**Fix.** Add tool `find_proteins_batch(genes: list[str], organism_taxon=None,
reviewed=None, limit_per_gene=5)`: resolves each gene via the existing reviewed-
first single-gene query **concurrently** (`asyncio.gather`), returning
`{by_gene: {GENE: [accessions]}, resolved: [...], unresolved_genes: [...],
count, elapsed_ms, cached}`, with `next_commands` fanning out
`get_protein`/`get_protein_features` on the resolved accessions. The canonical
"domains for PNKP and NAA10" task drops from ~11 s (two sequential lookups) to
~one cold round-trip. Surfacing `unresolved_genes` also kills another false-empty
edge. The delicate single `find_proteins` path is **untouched** (risk isolation).

**Files:** `mcp/tools/proteins.py` (register), `services/sparql_service.py`
(`find_proteins_batch`), `mcp/schemas.py` (FIND_PROTEINS_BATCH_SCHEMA),
`mcp/next_commands.py` (`after_find_proteins_batch`), `mcp/capabilities.py` +
`mcp/resources.py` (list it). **Tests:** `test_service_and_tools.py`,
`test_structured_output.py`, plus a live integration case.

### P1b — Leaner `get_protein_features` default

**Why.** ~18 of 35 features for a domains task were `beta_strand`/`turn`/`helix`
secondary structure — pure noise unless asked for.

**Fix.** Add `include_secondary_structure: bool = False`. When `False` **and** the
caller didn't explicitly name a secondary-structure type in `feature_types`,
exclude `{beta_strand, helix, turn}` from `features`, `count`, and `total`, and
disclose them under `excluded_secondary_structure: {count, types, hint:"pass
include_secondary_structure=true to include"}` (never silent). Explicit
`feature_types=["helix"]` still returns them (explicit beats default). Roughly
halves the most common feature payload — aligns with Anthropic's "trim rarely-used
fields from the default; expose a verbosity switch."

**Files:** `mcp/tools/proteins.py` (param + description),
`services/sparql_service.py` (`get_features` filtering),
`services/constants.py` (`SECONDARY_STRUCTURE_TYPES`), `mcp/schemas.py`. **Tests:**
`test_service_and_tools.py`.

## Risks & mitigations

- *Batch tool surface creep* → one well-described, namespaced tool with clear
  `next_commands`; single `find_proteins` left untouched.
- *name_contains behavior change* → strictly higher recall; single-word unchanged;
  documented + live-verified.
- *Features fetch-at-cap cost* → bound on one accession; `limit` "does not alter
  QLever's plan" (existing builder note), so fetching 1000 vs 200 rows is the same
  plan.
- *Extra count queries (variants/find_proteins)* → variants count is concurrent
  (no added wall-clock); find_proteins count runs only on truncation.

## Success criterion

Re-running the v0.7.0 tester protocol: F1–F8 each resolved with evidence, the
canonical multi-gene task feels fast, and no new silent/misleading edge — i.e. a
re-rated **> 9.5 / 10**.
